# Myelin

**연결성 기반 동적 정밀도 할당 (Connectivity-Aware Dynamic Precision Allocation)**

작성일: 2026-07-23 · 상태: 기획 확정, 구현 착수 전

---

## 1. 한 줄 요약

트랜스포머의 각 채널이 갖는 **연결성**을 학습 중 실시간으로 측정하고, 그 신호에 따라 채널별 저장 비트폭을 동적으로 늘리고 줄이는 학습 프레임워크.

핵심 주장: **어디에 비트를 더 줄지는 사람이 손으로 정할 게 아니라, 모델이 학습 중에 스스로 정해야 한다.**

---

## 2. 배경 및 문제의식

### 2.1 출발점

현재 모든 모델은 파라미터 타입을 fp32/fp16 등으로 **전역 고정**해두고 값만 학습시킨다. 그런데 모든 파라미터가 동등하게 중요하지는 않다. 어떤 자리는 아주 세밀한 캘리브레이션이 필요하고, 어떤 자리는 거칠어도 무방하다. 타입이 고정되어 있으면 이 차이를 표현할 방법이 없다.

### 2.2 생물학적 직관

뇌의 시냅스는 균일하지 않다. 자주 쓰이고 많이 연결된 경로일수록 강화되고, 수초(myelin)가 두꺼워져 신호 전달이 정밀하고 빨라진다. 덜 쓰이는 경로는 약한 상태로 남는다. **자원이 연결성에 비례해 배분되는 구조**다.

프로젝트명 Myelin은 여기서 왔다.

### 2.3 기존 접근과 무엇이 다른가

| 구분 | 기존 양자화 | Myelin |
|---|---|---|
| 시점 | 학습 완료 **후** 압축 | 학습 **중** 배분 |
| 배분 기준 | 균일하거나 사람이 튜닝한 휴리스틱 | 연결성 신호 |
| 가변성 | 한 번 정하면 고정 | 주기적 재배분 (증가/감소 모두) |

이건 사후 압축(PTQ)이 아니다. 학습 과정 자체에 정밀도 배분이 편입된다.

---

## 3. 핵심 가설

> **H1.** 채널별 연결성 지표는 해당 채널의 양자화 민감도를 예측한다.
>
> **H2.** 동일한 평균 비트 예산에서, 연결성 기반 배분은 무작위 배분보다 낮은 loss를 달성한다.
>
> **H3.** 동일한 평균 비트 예산에서, 연결성 기반 배분은 손튜닝 휴리스틱(llama.cpp k-quant)에 필적하거나 능가한다.

H2가 이 프로젝트의 최소 성공 조건이다. H2가 기각되면 아이디어 자체가 틀린 것이므로 빠르게 접는다.

## 4. 기술 설계

### 4.1 비트 플레인 분해 — 저장 포맷이 곧 메커니즘

가중치를 MSB부터 한 장씩 쌓이는 **평면(plane)** 으로 저장한다.

```
채널 A: [P0][P1][P2]           → 3비트
채널 B: [P0][P1][P2][P3][P4]   → 5비트
```

- "상한 3비트" = 앞의 세 평면만 읽는다
- 상한을 4로 올림 = 평면 하나 더 로드. **재양자화 없음, 저장 포맷 변경 없음**
- 배분 변경이 채널별 포인터 이동 수준으로 저렴해진다

**필수 조건:** 각 평면은 직전까지의 **잔차를 half step으로 쪼개는 방식**이어야 한다. min-max로 스케일을 다시 잡는 일반 per-channel 양자화는 비트 수가 바뀌면 스케일도 바뀌어 중첩이 깨진다.

**대가:** 잔차 방식은 특정 k에서의 최적 k비트 양자화보다 근소하게 나쁘다. *성장 가능성을 얻는 대가로 각 지점의 최적성을 조금 내준다.* (Matryoshka Quantization, Any-Precision LLM에서 이 격차를 다루며, 크지 않다는 것이 대체적 결론)

### 4.2 연결성 신호

두 가지 해석이 가능하다.

- **구조적 연결성** — 해당 뉴런의 in/out 가중치 norm 총합. "얼마나 강하게 이어져 있나"
- **동적 연결성** — 어텐션 가중치가 그 헤드/차원으로 얼마나 몰리나. "얼마나 실제로 쓰이나"

**제안:** 두 신호의 곱 `(가중치 norm) × (활성 흐름)`. → **§9 미결정 1번 참조**

**측정 위치 (확정):** 양자화된 값이 아니라 **shadow weight(fp32 원본)** 에서 측정한다.

이유: 비트를 많이 받은 채널 → 양자화 노이즈 감소 → 그래디언트가 깨끗해짐 → 더 중요해 보임 → 또 받음. 이 **자기실현적 루프**를 끊기 위함. STE 학습에서 어차피 fp32 원본을 들고 있으므로 추가 비용이 없다.

### 4.3 재배분 루프

**예산 구조: 제로섬.** 평균 비트 상한이 고정이므로, 누구에게 주는 것은 누구에게서 뺏는 것이다.

**축소를 허용하는 이유 (확정 사항):**

1. **신호가 학습 중 변한다.** 초기 측정값은 초기화 근처라 노이즈에 가깝다. 학습이 진행되며 어떤 헤드는 사실상 no-op이 되고(attention sink), 반대로 늦게 outlier feature로 튀어나와 중요해지는 차원이 생긴다.
2. **제로섬이므로 선점 문제가 생긴다.** 단조 증가만 허용하면 초반에 예산이 소진되고, 나중에 진짜 중요해진 채널에 줄 비트가 없다. "먼저 온 사람이 다 가져간다."
3. **자기실현 루프 차단.** §4.2의 피드백 고리는 회수 경로가 있어야 끊긴다.

**루프 사양:**

| 항목 | 값 | 근거 |
|---|---|---|
| 재배분 주기 | 500 스텝 | 매 스텝 재배분 시 유효 가중치가 점프해 학습 불안정 |
| 신호 평활 | EMA | 순간 노이즈 억제 |
| 재배분 폭 `k%` | 코사인 감쇠 | 초반 탐색 → 후반 고정. RigL과 동일 구조 |
| 비트 하한 | 1~2비트 | 0비트는 배분이 아니라 pruning. 죽은 채널은 신호가 안 나와 부활 불가 |
| 불감대 | 회수/지급 문턱 분리 | 컷라인 근처 채널이 매 주기 자리바꿈하는 것 방지 |

**중요:** 비트 플레인 구조 덕분에 **학습 중 축소는 실제 데이터 손실이 아니다.** 평면은 shadow weight에서 언제든 재생성되므로, 회수는 "이 평면 안 읽음" 플래그 토글이고 반환도 공짜다. 되돌릴 수 없는 결정은 **배포 시점 단 한 번**뿐. 회수 허용의 리스크가 통념보다 훨씬 작은 이유.

### 4.4 학습/추론 구조 — CPU를 정식 타깃으로

**왜 CPU인가 (GPU 대비 우위):**

1. **비트 단위 자유도.** GPU 텐서 코어는 정해진 타입만 먹어 3비트·5비트 같은 폭이 표현 불가. CPU는 시프트로 직접 패킹하므로 채널별 임의 비트폭이 실제로 가능하다.
2. **병목의 방향.** CPU 추론은 메모리 대역폭에 막힌다. 비트를 줄이면 트래픽이 줄고 속도가 바로 따라온다. GPU에서 int4가 기대만큼 안 빨라지는 건 이미 연산에 묶여 있기 때문.
3. **학습=배포 일치.** CPU에서는 forward에 **실제 비트 플레인 커널**을 그대로 꽂을 수 있다. 학습 중 forward와 배포 forward가 비트 단위로 동일해져, QAT의 고질적 문제인 "학습 땐 됐는데 실제 커널에서 달라지는" 구간이 사라진다. 흉내가 아니라 진짜가 된다.
4. **배포 환경과 일치.** 온프렘 노드(Ryzen 기반)와 그림이 맞는다.

**연산 분리:**

```
forward  → 패킹된 비트 플레인 커널 (Rust 대상)
backward → 표준 fp32 matmul (STE로 shadow weight에 직행)
```

활성 그래디언트는 full precision이라 패킹 커널이 안 쓰인다. **Rust로 짤 대상이 forward 하나로 좁혀지는 근거.**

**정직한 한계:** AVX-512/AMX도 결국 레인이 균일해야 빠르다. 섞인 비트폭은 연산 직전 공통 폭으로 언패킹된다. → **이득은 산술이 아니라 메모리에서 나온다.** 실험 시 "FLOPS는 그대로인데 왜 빨라졌나"에서 헤매지 않으려면 이걸 전제하고 시작해야 한다. (참고: Zen 3에는 AVX-512도 AMX도 없어 fp32가 기본)

**규모 타당성:**

- 학습 FLOPs ≈ 6 × 파라미터 × 토큰
- d_model 192, 4층 → 1~3M 파라미터
- 한국어 토큰 2~5천만 → 약 10¹⁴ FLOPs
- Zen 3 8코어 fp32 matmul 실측 30~50 GFLOPS → **실험 1회당 2시간 내외**
- 데이터는 한국어 위키 덤프 하나로 충당

이 프로젝트가 답할 질문은 "큰 모델 1회"가 아니라 **"작은 모델 수십 회"** 로 답하는 종류다. CPU가 오히려 맞다.

## 5. 기존 연구와의 위치

| 연구 | 하는 것 | 안 하는 것 = Myelin의 자리 |
|---|---|---|
| **HAWQ** 계열 | Hessian으로 레이어별 민감도 측정, mixed precision 배분 | 학습 후 1회 결정 후 고정. 레이어 단위 |
| **Matryoshka Quant / Any-Precision LLM** | 중첩 가능한 비트 포맷 자체를 설계 | **누구에게 다음 평면을 줄지**는 균일 또는 사후 휴리스틱 |
| **llama.cpp k-quant** | 텐서별로 비트를 다르게 배분 (V proj, FFN down에 더 많이) | 배분이 **사람이 실험해서 정한 휴리스틱**. 원리적 기준 없음 |
| **RigL** | 동적 희소 학습의 주기적 재배분 스케줄 | 대상이 연결의 존재 여부이지 정밀도가 아님 |
| **Stripes / BitFusion** | 비트 시리얼 하드웨어 데이터패스 | 비트폭을 정해주는 알고리즘은 외부 문제 |

**빈자리:** 중첩 포맷은 이미 있고, 비트 시리얼 하드웨어도 있고, k-quant는 "비트를 다르게 주면 좋다"를 이미 실증했다. **비어 있는 건 "그럼 어떤 원리로 배분할 것인가"** 다. 검증 부담이 크게 줄어드는 유리한 위치.

## 6. 실행 계획

### Phase 1 — PyTorch / CPU (현재)

미니 GPT + 비트 플레인 양자화 레이어 + 연결성 트래커 + 재배분 루프.

- 전부 CPU에서 동작
- forward의 양자화 부분은 **Rust 교체 가능한 형태로 인터페이스 분리**
- 목표: H2 검증. 아이디어의 생사가 여기서 결정된다

### Phase 2 — Rust 커널 (PyO3 하이브리드)

```
Rust  : 비트 플레인 패킹, 채널별 가변 비트 matmul, 캐시 레이아웃
Python: 학습 루프, 재배분 로직 (500스텝당 1회 · 채널별 요약 통계 · 성능 무관)
```

HuggingFace `tokenizers`와 동일한 구조.

**Rust를 Phase 1에 넣지 않는 이유:** Phase 1은 fake quant라 shadow weight가 전부 fp32이고 "3비트"는 반올림 산술일 뿐이다. **패킹할 비트가 아직 존재하지 않는다.** 여기에 Rust를 넣으면 느린 PyTorch를 Rust로 다시 쓰는 셈.

**학습 전체를 Rust로 옮기지 않는 이유:** autograd를 마주친다. Candle은 추론 중심, Burn은 autograd가 있으나 미성숙, 직접 구현은 수개월. 그 수개월이 "연결성 배분이 무작위보다 나은가"에 아무 답도 주지 않는다. 아이디어가 틀렸을 경우 인프라만 남는다.

### Phase 3 — Verilog RTL (비트 시리얼 가속기)

비트 플레인 = **비트 시리얼**이다. 연산 시간이 비트 수에 선형 비례하므로 "쓴 만큼만 낸다"가 그대로 성립한다. 채널별 가변 비트폭을 네이티브로 먹는 데이터패스는 결국 실리콘에서만 제대로 나온다.

**Phase 2의 Rust 커널이 RTL 검증의 골든 모델로 그대로 재사용된다.** 같은 입력을 넣고 사이클 단위 비교. 버려지는 코드가 아니라 필수 자산.

> Myelin의 진짜 종착지는 프레임워크가 아니라 여기일 가능성이 높다. "이 알고리즘을 위한 가속기"는 논문 한 편보다 강한 이야기다.

## 7. 실험 설계

**핵심 실험:**

```
전 채널 2비트에서 시작
  → 연결성으로 순위 매김
  → 상위부터 평면 하나씩 추가
  → 평균 비트 예산 도달 시 정지
  → 이후 500스텝마다 재배분 (회수 포함, k% 코사인 감쇠)
```

**Baseline:**

| 우선순위 | Baseline | 검증 대상 |
|---|---|---|
| 1차 | **같은 예산 · 무작위 순서로 평면 추가** | H2. 가장 깨끗한 대조군 |
| 2차 | llama.cpp k-quant 배분 | H3 |
| — | ~~균일 4비트~~ | 비트 플레인 도입으로 비교 대상으로서 약해짐 |

**실험 매트릭스:** 예산 4~5 수준 × 배분 전략 3종 × 시드 3개 = **40회 이상**

**병렬 실행 주의:**
- PyTorch CPU는 텐서가 작을 때 연산당 오버헤드가 크다 → **배치를 충분히 키울 것**
- `torch.set_num_threads`를 실험 개수에 맞춰 명시적으로 분배할 것. 안 하면 프로세스끼리 코어를 뺏어 전부 느려진다
- 코어 분할 병렬 시 하루 내 전량 소화 가능

**측정 항목:** loss / perplexity, 실제 메모리 트래픽, wall-clock, 배분 궤적(어떤 채널이 언제 비트를 얻고 잃었나), 비트 시리얼 vs AVX-512 int8 **크로스오버 지점**

## 8. 리스크 및 정직한 한계

| 리스크 | 내용 | 대응 |
|---|---|---|
| **잔차 방식 손실** | 중첩성을 위해 각 지점의 최적성을 일부 포기 | 선행 연구상 격차는 작음. Phase 1에서 정량화 |
| **속도 이득이 안 나올 수 있음** | 이득은 메모리에서만 나오고, 비트 시리얼은 상수 항이 나빠 평균 비트가 낮을 때만 이김 | 크로스오버 지점 자체를 실험 결과물로 삼음 |
| **재배분 불안정** | 유효 가중치 점프로 학습 진동 | EMA + 주기 제한 + 불감대 + k 감쇠 |
| **H2 기각** | 연결성이 무작위보다 낫지 않음 | **빠르게 접는다.** 단, "왜 사람 직관이 더 나았는가"는 그 자체로 건질 결과 |

---

## 9. 미결정 사항

1. **연결성 정의 확정** — `가중치 norm × 활성 흐름`(제안) vs 순수 구조적 연결성만
2. **토크나이저** — 한국어 BPE 직접 학습 vs 기존 토크나이저 사용
3. **어휘 크기** — 임베딩 파라미터 비중을 크게 좌우. 1~3M 예산 내에서 결정 필요

> 3번은 2번에 종속. 1번은 Phase 1 착수 전 확정 필요.
